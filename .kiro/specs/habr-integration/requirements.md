# Документ требований

## Введение

Расширение существующего сервиса «Pikabu Topic Analyzer» поддержкой парсинга и анализа статей и комментариев с сайта habr.com. Система должна поддерживать три режима работы: только Pikabu, только Habr, Pikabu + Habr (комбинированный). Для Habr парсятся статьи из потоков (flows): management, top_management, marketing. Habr доступен без прокси и без аутентификации. Пайплайн анализа (парсинг → разбиение на чанки → LLM-анализ → агрегация) остаётся единым для обоих источников. Отчёты должны указывать, из какого источника (или источников) были получены данные.

## Глоссарий

- **Система** — серверная часть приложения (FastAPI backend)
- **SPA** — одностраничное веб-приложение (Vue 3 + TypeScript), через которое Пользователь взаимодействует с Системой
- **Парсер_Pikabu** — существующий модуль Системы, отвечающий за извлечение данных с pikabu.ru
- **Парсер_Habr** — новый модуль Системы, отвечающий за извлечение статей и комментариев с habr.com
- **Анализатор** — модуль Системы, отвечающий за многоэтапный анализ собранных данных с помощью LLM
- **Источник** — платформа, с которой собираются данные: Pikabu, Habr или обе одновременно
- **Режим_источника** — выбранный пользователем режим парсинга: «pikabu» (только Pikabu), «habr» (только Habr), «both» (Pikabu + Habr)
- **Поток_Habr** — раздел (flow) на habr.com, содержащий статьи по определённой тематике (например, management, top_management, marketing)
- **Статья_Habr** — публикация на habr.com, содержащая заголовок, тело, дату публикации, рейтинг, количество комментариев и URL
- **Тема** — предопределённое сообщество или тег на pikabu.ru, либо поток на habr.com, доступный для выбора из списка
- **Список_тем** — перечень доступных тем, загружаемый с pikabu.ru и/или habr.com в зависимости от выбранного Режима_источника
- **Чанк** — часть собранных данных, размер которой не превышает допустимый лимит контекстного окна LLM
- **Отчёт** — итоговый структурированный результат анализа с указанием использованных источников
- **Пользователь** — человек, использующий SPA для выбора темы, запуска анализа и просмотра отчётов
- **Хранилище** — база данных PostgreSQL

## Требования

### Требование 1: Выбор режима источника данных

**Пользовательская история:** Как пользователь, я хочу выбирать источник данных (Pikabu, Habr или оба), чтобы анализировать контент с нужной мне платформы.

#### Критерии приёмки

1. THE SPA SHALL отображать на странице выбора темы селектор Режима_источника с тремя вариантами: «Pikabu», «Habr», «Pikabu + Habr»
2. WHEN страница выбора темы загружается впервые, THE SPA SHALL устанавливать Режим_источника в значение «pikabu» по умолчанию
3. WHEN пользователь переключает Режим_источника, THE SPA SHALL обновлять Список_тем в соответствии с выбранным режимом
4. WHEN Режим_источника установлен в «pikabu», THE SPA SHALL отображать только темы с pikabu.ru
5. WHEN Режим_источника установлен в «habr», THE SPA SHALL отображать только потоки с habr.com
6. WHEN Режим_источника установлен в «both», THE SPA SHALL отображать темы с pikabu.ru и потоки с habr.com с визуальным разделением по источнику

### Требование 2: Получение списка потоков Habr

**Пользовательская история:** Как пользователь, я хочу видеть список доступных потоков Habr, чтобы выбрать интересующий меня поток для анализа.

#### Критерии приёмки

1. THE Система SHALL предоставлять предопределённый список потоков Habr: management, top_management, marketing
2. WHEN пользователь запрашивает список тем с Режимом_источника «habr» или «both», THE Система SHALL возвращать потоки Habr в формате, совместимом с существующим Списком_тем
3. THE Система SHALL формировать URL каждого потока Habr по шаблону: https://habr.com/ru/flows/{flow_name}/articles/
4. THE Система SHALL присваивать каждому потоку Habr уникальный идентификатор с префиксом «habr_» для отличия от тем Pikabu

### Требование 3: Парсинг статей Habr из потоков

**Пользовательская история:** Как пользователь, я хочу, чтобы система собирала статьи из выбранного потока Habr, чтобы анализ был основан на актуальных данных.

#### Критерии приёмки

1. WHEN парсинг запущен для потока Habr, THE Парсер_Habr SHALL извлекать статьи из страницы потока по URL https://habr.com/ru/flows/{flow_name}/articles/
2. THE Парсер_Habr SHALL извлекать для каждой статьи: заголовок, тело, дату публикации, рейтинг, количество комментариев и URL
3. THE Парсер_Habr SHALL поддерживать пагинацию страниц потока для загрузки всех доступных статей за запрошенный период
4. WHEN статья на странице потока имеет дату публикации старше запрошенного периода, THE Парсер_Habr SHALL прекращать пагинацию (early-exit), так как статьи отсортированы по дате от новых к старым
5. THE Парсер_Habr SHALL выполнять HTTP-запросы к habr.com без использования прокси
6. THE Парсер_Habr SHALL выполнять HTTP-запросы к habr.com без аутентификации
7. IF habr.com возвращает ошибку HTTP 429, THEN THE Парсер_Habr SHALL приостанавливать запросы на 60 секунд и повторять попытку
8. IF habr.com возвращает ошибку HTTP 5xx, THEN THE Парсер_Habr SHALL повторять запрос до 3 раз с интервалом 10 секунд между попытками

### Требование 4: Парсинг комментариев к статьям Habr

**Пользовательская история:** Как пользователь, я хочу, чтобы система собирала комментарии к статьям Habr, чтобы анализ учитывал мнения читателей.

#### Критерии приёмки

1. WHEN статья Habr успешно спарсена, THE Парсер_Habr SHALL извлекать все комментарии к данной статье
2. THE Парсер_Habr SHALL извлекать для каждого комментария: текст, дату публикации и рейтинг
3. IF парсинг комментариев к отдельной статье завершается ошибкой, THEN THE Парсер_Habr SHALL пропускать комментарии данной статьи и продолжать парсинг следующих статей

### Требование 5: Хранение данных Habr

**Пользовательская история:** Как пользователь, я хочу, чтобы спарсенные данные Habr сохранялись в базе данных, чтобы повторный анализ не требовал повторного парсинга.

#### Критерии приёмки

1. WHEN парсинг потока Habr завершён, THE Система SHALL сохранять все извлечённые статьи и комментарии в Хранилище
2. THE Система SHALL хранить статьи Habr и посты Pikabu в единой таблице постов, различая их по идентификатору источника
3. THE Система SHALL хранить комментарии Habr и комментарии Pikabu в единой таблице комментариев
4. THE Система SHALL хранить метку времени последнего парсинга для каждого потока Habr
5. WHEN пользователь запрашивает анализ потока Habr, для которого данные были спарсены менее 24 часов назад, THE Система SHALL использовать кэшированные данные из Хранилища без повторного парсинга

### Требование 6: Комбинированный анализ Pikabu + Habr

**Пользовательская история:** Как пользователь, я хочу анализировать данные с обеих платформ одновременно, чтобы получить более полную картину обсуждений по теме.

#### Критерии приёмки

1. WHEN Режим_источника установлен в «both» и пользователь выбрал тему Pikabu и поток Habr, THE Система SHALL запускать парсинг обоих источников
2. WHEN парсинг обоих источников завершён, THE Анализатор SHALL объединять данные из обоих источников в единый набор для разбиения на Чанки
3. THE Анализатор SHALL обрабатывать объединённые данные через тот же пайплайн: разбиение на чанки → LLM-анализ каждого чанка → агрегация результатов
4. WHILE комбинированный парсинг выполняется, THE SPA SHALL отображать прогресс с указанием текущего источника (например, «Загрузка данных с Pikabu...», «Загрузка данных с Habr...»)

### Требование 7: Указание источника в отчётах

**Пользовательская история:** Как пользователь, я хочу видеть в отчёте, из какого источника были получены данные, чтобы понимать контекст анализа.

#### Критерии приёмки

1. THE Отчёт SHALL содержать поле с указанием использованных источников данных (Pikabu, Habr или оба)
2. THE SPA SHALL отображать в заголовке Отчёта информацию об использованных источниках
3. WHEN Отчёт содержит данные из обоих источников, THE SPA SHALL отображать ссылки на оригинальные посты с указанием платформы (Pikabu или Habr)
4. THE Система SHALL сохранять информацию об использованных источниках в Хранилище вместе с Отчётом

### Требование 8: Расширение REST API для поддержки Habr

**Пользовательская история:** Как разработчик frontend, я хочу, чтобы REST API поддерживал работу с несколькими источниками данных, чтобы SPA мог корректно отображать и управлять данными из разных платформ.

#### Критерии приёмки

1. THE Система SHALL принимать параметр source в endpoint GET /api/topics со значениями «pikabu», «habr» или «both» для фильтрации Списка_тем по источнику
2. WHEN параметр source не указан в запросе GET /api/topics, THE Система SHALL возвращать только темы Pikabu для обратной совместимости
3. THE Система SHALL принимать параметр source в endpoint POST /api/analysis/start для указания Режима_источника при запуске анализа
4. WHEN Режим_источника установлен в «both», THE Система SHALL принимать в POST /api/analysis/start идентификаторы тем для обоих источников
5. IF запрос POST /api/analysis/start содержит Режим_источника «habr» или «both» без указания идентификатора потока Habr, THEN THE Система SHALL возвращать HTTP-ответ с кодом 400 и описанием ошибки

### Требование 9: Парсинг HTML-страниц Habr

**Пользовательская история:** Как разработчик, я хочу, чтобы парсер Habr корректно извлекал данные из HTML-страниц, чтобы обеспечить надёжный сбор информации.

#### Критерии приёмки

1. THE Парсер_Habr SHALL извлекать статьи из HTML-страницы потока Habr, используя CSS-селекторы для элементов списка статей
2. THE Парсер_Habr SHALL извлекать комментарии из HTML-страницы статьи Habr, используя CSS-селекторы для элементов комментариев
3. THE Парсер_Habr SHALL корректно обрабатывать HTML-страницы Habr, содержащие кириллический текст в кодировке UTF-8
4. THE Pretty_Printer SHALL форматировать извлечённые данные статей Habr в структурированный словарь с ключами: title, body, published_at, rating, comments_count, url, source_id
5. FOR ALL валидных HTML-страниц потока Habr, парсинг с последующим форматированием и повторным парсинг SHALL возвращать эквивалентный набор данных (свойство round-trip)
