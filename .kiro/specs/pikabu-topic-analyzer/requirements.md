# Документ требований

## Введение

Сервис «Pikabu Topic Analyzer» предназначен для анализа контента на сайте pikabu.ru по выбранной пользователем теме. Пользователь выбирает тему из списка предопределённых сообществ/тегов, доступных на pikabu.ru. Система парсит посты и комментарии за последний месяц, анализирует собранные данные с помощью LLM (Google Gemini API) в несколько этапов (разбиение на чанки, анализ каждого чанка, агрегация результатов) и формирует структурированный отчёт о часто обсуждаемых темах, проблемах пользователей и трендовых дискуссиях. Многоэтапный анализ необходим, так как объём постов и комментариев для популярных тем может быть огромным и не помещаться в контекстное окно LLM. Backend реализован на Python (FastAPI), frontend — SPA (React или Vue), данные хранятся в PostgreSQL.

## Глоссарий

- **Система** — серверная часть приложения Pikabu Topic Analyzer (FastAPI backend)
- **SPA** — одностраничное веб-приложение (React или Vue), через которое пользователь взаимодействует с Системой
- **Парсер** — модуль Системы, отвечающий за извлечение данных (постов и комментариев) с сайта pikabu.ru с помощью BeautifulSoup или Scrapy
- **Анализатор** — модуль Системы, отвечающий за многоэтапный анализ собранных данных с помощью Google Gemini API
- **Gemini_API** — Google Gemini API, используемый для интеллектуального анализа текстовых данных
- **Тема** — предопределённое сообщество или тег на сайте pikabu.ru, доступное для выбора из списка
- **Список_тем** — перечень доступных тем (сообществ/тегов), загружаемый с pikabu.ru для отображения пользователю
- **Чанк** — часть собранных данных (постов и комментариев), размер которой не превышает допустимый лимит контекстного окна Gemini API
- **Частичный_результат** — результат анализа одного чанка данных, содержащий промежуточные выводы
- **Отчёт** — итоговый структурированный результат анализа, сформированный путём агрегации всех частичных результатов, содержащий часто обсуждаемые темы, проблемы пользователей и трендовые дискуссии
- **Пользователь** — человек, использующий SPA для выбора темы, запуска анализа и просмотра отчётов
- **Хранилище** — база данных PostgreSQL, используемая для хранения спарсенных данных и результатов анализа

## Требования

### Требование 1: Выбор темы для анализа из списка

**Пользовательская история:** Как пользователь, я хочу выбрать тему из списка доступных сообществ/тегов на Pikabu, чтобы запустить анализ контента по интересующей меня области.

#### Критерии приёмки

1. WHEN пользователь открывает страницу выбора темы, THE Система SHALL загружать Список_тем доступных сообществ и тегов с pikabu.ru
2. THE SPA SHALL отображать Список_тем в виде прокручиваемого списка с возможностью поиска по названию
3. WHEN пользователь вводит текст в поле поиска, THE SPA SHALL фильтровать Список_тем, отображая только темы, содержащие введённый текст в названии
4. WHEN пользователь выбирает тему из Списка_тем, THE SPA SHALL отображать базовую информацию о выбранной теме (название, количество подписчиков)
5. IF загрузка Списка_тем с pikabu.ru завершается ошибкой, THEN THE Система SHALL возвращать сообщение об ошибке с указанием причины недоступности списка тем


### Требование 2: Запуск парсинга по запросу

**Пользовательская история:** Как пользователь, я хочу запускать парсинг по нажатию кнопки, чтобы контролировать момент сбора данных.

#### Критерии приёмки

1. WHEN пользователь нажимает кнопку «Начать анализ», THE Система SHALL запускать процесс парсинга для выбранной из списка темы
2. WHILE парсинг выполняется, THE SPA SHALL отображать индикатор прогресса с текущим статусом операции
3. WHILE парсинг выполняется, THE SPA SHALL блокировать повторное нажатие кнопки «Начать анализ» для той же темы
4. IF парсинг прерывается из-за сетевой ошибки, THEN THE Система SHALL сохранять уже собранные данные и уведомлять пользователя о частичном результате

### Требование 3: Парсинг постов и комментариев

**Пользовательская история:** Как пользователь, я хочу, чтобы система собирала все посты и комментарии за последний месяц, чтобы анализ был основан на актуальных данных.

#### Критерии приёмки

1. WHEN парсинг запущен, THE Парсер SHALL извлекать все посты по выбранной теме, опубликованные за последние 30 дней
2. WHEN парсинг запущен, THE Парсер SHALL извлекать все комментарии к каждому найденному посту
3. THE Парсер SHALL сохранять для каждого поста: заголовок, текст, дату публикации, рейтинг и количество комментариев
4. THE Парсер SHALL сохранять для каждого комментария: текст, дату публикации и рейтинг
5. IF pikabu.ru возвращает ошибку HTTP 429 (слишком много запросов), THEN THE Парсер SHALL приостанавливать запросы на 60 секунд и повторять попытку
6. IF pikabu.ru возвращает ошибку HTTP 5xx, THEN THE Парсер SHALL повторять запрос до 3 раз с интервалом 10 секунд между попытками

### Требование 4: Хранение спарсенных данных

**Пользовательская история:** Как пользователь, я хочу, чтобы спарсенные данные сохранялись в базе данных, чтобы повторный анализ не требовал повторного парсинга.

#### Критерии приёмки

1. WHEN парсинг завершён, THE Система SHALL сохранять все извлечённые посты и комментарии в Хранилище
2. WHEN пользователь запрашивает анализ темы, для которой данные были спарсены менее 24 часов назад, THE Система SHALL использовать кэшированные данные из Хранилища без повторного парсинга
3. WHEN пользователь запрашивает анализ темы, для которой данные были спарсены более 24 часов назад, THE Система SHALL запускать повторный парсинг
4. THE Система SHALL хранить метку времени последнего парсинга для каждой темы


### Требование 5: Многоэтапный анализ контента с помощью Gemini API

**Пользовательская история:** Как пользователь, я хочу получить интеллектуальный анализ собранных данных, чтобы понять основные тенденции и проблемы в выбранной теме, даже если объём данных очень большой.

#### Критерии приёмки

1. WHEN парсинг завершён, THE Анализатор SHALL разбивать все собранные посты и комментарии на Чанки, размер каждого из которых не превышает допустимый лимит контекстного окна Gemini_API
2. WHEN данные разбиты на Чанки, THE Анализатор SHALL последовательно отправлять каждый Чанк в Gemini_API для анализа и получать Частичный_результат для каждого Чанка
3. THE Анализатор SHALL формировать в каждом Частичном_результате: список обсуждаемых тем с количеством упоминаний, список проблем пользователей и список активных дискуссий
4. WHEN все Чанки проанализированы, THE Анализатор SHALL отправлять все Частичные_результаты в Gemini_API для агрегации в единый итоговый Отчёт
5. THE Анализатор SHALL формировать в итоговом Отчёте: объединённый список часто обсуждаемых тем с суммарным количеством упоминаний, консолидированный список проблем и болевых точек пользователей, список трендовых дискуссий, отсортированных по активности
6. IF Gemini_API возвращает ошибку при анализе Чанка, THEN THE Система SHALL повторять запрос до 3 раз с экспоненциальной задержкой между попытками
7. IF Gemini_API недоступен после всех повторных попыток, THEN THE Система SHALL уведомлять пользователя о невозможности выполнить анализ и сохранять спарсенные данные для последующего анализа
8. IF агрегация Частичных_результатов завершается ошибкой Gemini_API, THEN THE Система SHALL сохранять все Частичные_результаты и уведомлять пользователя о возможности повторить агрегацию

### Требование 6: Формирование и отображение отчёта

**Пользовательская история:** Как пользователь, я хочу видеть результаты анализа в виде структурированного отчёта, чтобы быстро понять ситуацию в выбранной теме.

#### Критерии приёмки

1. WHEN анализ завершён, THE SPA SHALL отображать Отчёт, содержащий три раздела: «Часто обсуждаемые темы», «Проблемы пользователей» и «Трендовые дискуссии»
2. THE SPA SHALL отображать в разделе «Часто обсуждаемые темы» список тем с кратким описанием и количеством упоминаний
3. THE SPA SHALL отображать в разделе «Проблемы пользователей» список проблем с описанием и примерами из комментариев
4. THE SPA SHALL отображать в разделе «Трендовые дискуссии» список дискуссий со ссылками на оригинальные посты на pikabu.ru
5. WHEN пользователь открывает ранее сгенерированный Отчёт, THE SPA SHALL отображать дату и время генерации Отчёта

### Требование 7: Хранение результатов анализа

**Пользовательская история:** Как пользователь, я хочу иметь доступ к ранее сгенерированным отчётам, чтобы отслеживать изменения в теме со временем.

#### Критерии приёмки

1. WHEN анализ завершён, THE Система SHALL сохранять Отчёт в Хранилище с привязкой к теме и метке времени
2. THE SPA SHALL отображать список ранее сгенерированных отчётов для выбранной темы, отсортированный по дате создания
3. WHEN пользователь выбирает отчёт из списка, THE SPA SHALL отображать полное содержимое выбранного Отчёта


### Требование 8: REST API

**Пользовательская история:** Как разработчик frontend, я хочу взаимодействовать с backend через REST API, чтобы обеспечить чёткое разделение ответственности между клиентом и сервером.

#### Критерии приёмки

1. THE Система SHALL предоставлять REST API endpoint GET /api/topics для получения Списка_тем доступных сообществ и тегов с pikabu.ru
2. THE Система SHALL предоставлять REST API endpoint POST /api/analysis/start для запуска парсинга и анализа по выбранной теме из Списка_тем
3. THE Система SHALL предоставлять REST API endpoint GET /api/analysis/status/{task_id} для получения статуса текущего парсинга и анализа
4. THE Система SHALL предоставлять REST API endpoint GET /api/reports/{topic_id} для получения списка отчётов по теме
5. THE Система SHALL предоставлять REST API endpoint GET /api/reports/{topic_id}/{report_id} для получения конкретного отчёта
6. THE Система SHALL возвращать ответы API в формате JSON
7. IF запрос содержит невалидные параметры, THEN THE Система SHALL возвращать HTTP-ответ с кодом 400 и описанием ошибки валидации

### Требование 9: Обработка больших объёмов данных при многоэтапном анализе

**Пользовательская история:** Как пользователь, я хочу, чтобы система корректно обрабатывала темы с очень большим количеством постов и комментариев, чтобы анализ был полным и точным независимо от объёма данных.

#### Критерии приёмки

1. THE Анализатор SHALL разбивать собранные данные на Чанки таким образом, чтобы размер каждого Чанка не превышал 80 процентов от лимита контекстного окна Gemini_API, оставляя запас для системного промпта и ответа
2. WHEN количество Частичных_результатов превышает лимит контекстного окна Gemini_API для агрегации, THE Анализатор SHALL выполнять агрегацию иерархически: сначала объединять Частичные_результаты в промежуточные группы, затем объединять промежуточные группы в итоговый Отчёт
3. THE Анализатор SHALL обрабатывать темы, содержащие до 10000 постов и соответствующие комментарии за последние 30 дней
4. WHILE многоэтапный анализ выполняется, THE SPA SHALL отображать прогресс анализа с указанием текущего этапа (разбиение на чанки, анализ чанка N из M, агрегация результатов)
